# 計畫問題與風險檢查

這份藍圖方向正確，但若要拿去創業、招工程師、找投資人或賣給店家，必須先修正以下風險。

## 1. 最大問題：範圍過大

原始構想包含 POS、QR 點餐、LINE、外送、電子發票、付款、庫存、會員、行銷、AI、連鎖總部、硬體、API、匿名 benchmark、競品價格監控、社群貼文產生。這是多條產品線，不是一個 MVP。

風險：

- 工程期拉長，第一版永遠出不來。
- 每個模組都做淺，賣點反而不可信。
- 客服與導入成本在還沒找到 PMF 前就爆炸。

修正：

- MVP 只服務外帶型小店：早餐、飲料、小吃、便當。
- 第一版只承諾「收銀 + 掃碼/LINE 點餐 + 出單 + 交班 + 基本報表 + 菜單匯入 + 基本離線」。
- 電子發票第一版先做串接與驗證路線，不要承諾全自建。
- 連鎖總部、完整庫存、AI 備料、匿名 benchmark 放到 P2/P3。

## 2. 定價可能低估成本

NT$399-599 可以成為切入價格，但若包含真人客服、電子發票、LINE、AI、雲端、遠端診斷與高可用同步，毛利會被快速吃掉。

風險：

- 小店客單低、問題多、客服成本高。
- 發票與付款問題一旦出錯，客服成本與信任損失都高。
- 低價方案若功能太完整，Pro 升級誘因不足。

修正：

- Starter 限制商品數、機台數、報表深度、客服 SLA。
- 電子發票列為 Basic 以上或獨立加購。
- AI 日報 Basic 可給摘要，進階建議放 Pro。
- 到府安裝、資料搬家、硬體設定收一次性費用。

## 3. 電子發票不能當一般功能處理

台灣電子發票涉及 MIG、Turnkey、字軌、上傳期限、作廢、折讓、B2B/B2C、載具、捐贈碼、稅務風險。2026 起 MIG 4.1 / Turnkey 3.2 相關要求會讓舊串接失效。

風險：

- 漏傳、重號、折讓錯誤會直接傷害店家信任。
- 自建 Turnkey 維運需要法遵、測試、異常處理能力。
- 離線開立發票尤其容易出問題。

修正：

- 把電子發票列為獨立服務與責任邊界。
- 第一版優先串成熟加值中心或合作夥伴。
- 必做發票健康儀表板與日結比對。
- 未完成法遵驗證前，不在銷售頁承諾「完整電子發票」。

## 4. 離線模式要重新定義

「斷網可用」很有殺傷力，但不能模糊承諾所有功能離線都正常。

風險：

- 行動支付通常需要網路或已授權流程。
- 載具驗證、發票上傳、線上點餐、LINE 通知不能完全離線。
- 多機台同時離線會造成序號、庫存、訂單衝突。

修正：

明確定義離線可用範圍：

- 可用：本機收銀、現金付款、列印、暫存訂單、同步佇列。
- 限制：行動支付、載具即時驗證、外送平台接單、雲端會員查詢。
- 必須提示：離線筆數、離線時間、待同步金額、發票風險。

## 5. AI 亮點要避免過度承諾

AI 是差異化，但早期資料量不足、店家資料品質不穩，不能一開始承諾準確備料、漲價、促銷收益預測。

風險：

- AI 建議錯，老闆會失去信任。
- 促銷與備料涉及毛利、浪費、供應鏈，資料不完整時容易誤導。
- LLM 摘要若沒有嚴格資料邊界，可能產生幻覺。

修正：

- 第一版 AI 定位為「營運摘要與異常提醒」。
- 所有 AI 建議附上依據數據，例如「因為 14 天平均銷量下降 18%」。
- 重要決策用「建議」而非「自動執行」。
- 備料先從規則與歷史平均開始，再逐步模型化。

## 6. 支付與 PCI 範圍需嚴格控管

若 POS 接觸完整卡號或敏感認證資料，PCI 範圍會大幅擴大。

風險：

- 合規成本超出新創能力。
- 資安事件會毀掉品牌。
- 支付合作商審查變慢。

修正：

- POS 不儲存、不處理、不傳輸完整卡號與 CVV。
- 刷卡由合格刷卡機或金流 SDK 處理。
- POS 只保存交易序號、授權碼、狀態與對帳資訊。
- 付款架構文件先於工程開發完成。

## 7. 多租戶與權限是高風險區

POS 系統會保存營收、會員、發票、員工操作、付款資料。一旦租戶隔離錯誤，A 店看到 B 店資料就是重大事故。

風險：

- Broken object-level authorization。
- 共用快取 key 洩漏。
- 內部客服權限過大。
- 匯出功能外洩大量個資。

修正：

- 所有資料表保留 tenant_id。
- 所有 query 必須 tenant-scoped。
- 快取、檔案、queue 都要 tenant namespace。
- 客服後台需要 impersonation audit。
- 匯出資料要二次驗證與稽核。

## 8. 產品承諾要避開「低價又全包」陷阱

「低月租 + 功能深度接近高價 POS + AI + 電子發票 + 不抽成 + 不綁硬體」很吸引人，但若對外講成全包，會造成不切實際期待。

修正後定位：

> 入門價低，成長功能可逐步加購；小店先解決接單和收銀，長大後再升級發票、庫存、會員、AI、連鎖。

## 9. 建議先驗證的 10 個問題

在正式開發前，至少訪談 30 家店並驗證：

1. 店家是否願意用自己的平板 / 手機做 POS？
2. 店家對 NT$399、NT$799、NT$1,299 的價格敏感度差異？
3. 電子發票是否是成交必要條件？
4. LINE 點餐是否比 QR 網頁更有成交力？
5. 店家是否願意自助建菜單，還是一定要人協助？
6. 尖峰時可接受的點餐/結帳步驟數？
7. 店家實際使用哪些付款方式？
8. 現在最痛的是接單、出單、發票、庫存、會員還是報表？
9. 願意為 AI 日報付費嗎？還是只把它當免費附加？
10. 是否願意從既有 POS 搬家？搬家最大的阻力是什麼？

## 10. 結論

這個產品有機會，但必須用「小店先成交，架構能長大」的方式做。

最小可打市場版本應該聚焦：

> 收銀、QR/LINE 點餐、出單、交班、基本報表、菜單匯入、LINE 日報、基本離線。

等到前 20-50 家店穩定營業，再擴展電子發票深度、庫存、會員、AI 促銷、連鎖總部。

## 目前實作差距（2026-05 對齊）

- **Persistence**: single `data/store.json` file，無真實 DB。Concurrency unsafe past PoC volume。
- **Sync worker**: 現為 in-process interval daemon，非獨立 process。Cloud reconcile path 缺失。
- **Invoice**: 僅 sandbox stubs。加值中心 / Turnkey adapter 未接線。`docs/high-risk-workstreams.md` 的 go-gate 未開。
- **Payment**: 手動現金 + 手動卡片記錄唯一。無 gateway SDK，無 PCI-compliant flow。
- **KDS**: HTTP poll endpoints 唯一。WebSocket/MQTT station channel 缺失。
- **Monitoring**: console.log 唯一。Prometheus/Grafana/Loki 缺失。
- **Auth**: bearer 32-byte hex 唯一。無 MFA，無 refresh token，無 device binding enforcement。
- **Multi-store HQ control, franchise audit, anonymous benchmark**: 未開始。

以上差距均屬 demo 階段已知限制；上線前須對齊 docs/high-risk-workstreams.md 的 go-gate。
