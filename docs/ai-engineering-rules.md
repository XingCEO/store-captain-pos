# AI 與工程品質硬規則

本文件是專案硬規範。所有 AI、工程師、產品、顧問、外包、代理人進場前都必須先讀。目的只有一個：禁止偷工減料、禁止省略關鍵細節、禁止留下豆腐渣工程與後補技術債。

## 1. 最高原則

> 不准用模糊文字換速度，不准用表面完成換真實可用。

任何產出都必須能回答：

- 誰使用？
- 為什麼做？
- 資料從哪裡來？
- 狀態怎麼流轉？
- 失敗怎麼處理？
- 誰有權限？
- 怎麼驗收？
- 上線後誰負責監控？

答不出來，就不是完成。

## 2. 禁止用語

以下用語不得單獨出現在規格、任務、設計、驗收或交付說明中：

- 等等
- 之類
- 相關
- 一些
- 基本上
- 視情況
- 後續補
- 先簡單做
- 暫時先這樣
- 應該可以
- 大概
- 有需要再做
- 以後優化

若必須使用，後面一定要接明確清單、範圍、條件或決策人。

錯誤：

```text
處理付款失敗等等狀況。
```

正確：

```text
付款失敗需處理：授權失敗、請款失敗、退款失敗、交易逾時、重複請求、金額不一致、網路中斷、PSP 回傳未知狀態。每種狀態需有錯誤碼、可重試規則、人工處理流程與 audit log。
```

## 3. 規格不可省略項

任何功能規格至少要包含：

| 項目 | 必填內容 |
| --- | --- |
| 目的 | 解決哪個店家問題 |
| 使用者 | 老闆、店長、收銀員、總部、客服、財務等 |
| 範圍 | 做什麼、不做什麼 |
| 狀態 | 所有狀態與狀態轉換 |
| 權限 | 誰能看、誰能改、誰能核准 |
| 資料 | 欄位、來源、保存位置、保存期限 |
| 例外 | 失敗、重試、人工補救、客服流程 |
| 稽核 | 哪些操作要留 log |
| 驗收 | 測試案例、手動 QA、上線門檻 |
| 風險 | 法規、金流、發票、個資、資料一致性 |

缺任一項，規格不得進開發。

## 4. 工程不可省略項

任何工程設計至少要明確寫出：

- API contract
- request / response schema
- error codes
- database tables / indexes
- idempotency key 規則
- tenant scope 規則
- permission checks
- audit log 欄位
- retry / timeout / dead letter 規則
- observability：log、metric、trace、alert
- migration / rollback 方式
- manual repair / admin tool 方式

不得只寫「做 API」、「串金流」、「接發票」、「做同步」。

## 5. POS 專案不可妥協項

以下項目不得被排到「以後再補」：

### 離線同步

- 本機寫入與 outbox enqueue 必須同 transaction。
- 雲端必須有 idempotency。
- 同 key 不同 payload 必須拒絕。
- 必須有 retry、backoff、dead letter、人工補救。
- 不得做 local SQLite 直接覆蓋 cloud PostgreSQL 的 CRUD 同步。

### 多租戶

- 所有查詢必須 tenant-scoped。
- 不得信任 client 傳來的 `tenant_id`。
- cache key、queue、file path 都必須帶 tenant context。
- 客服跨租戶操作必須 audit。

### 電子發票

- 發票狀態不可只靠文字備註。
- 必須有開立、上傳、失敗、重試、作廢、折讓、退貨狀態。
- 必須能對帳：訂單金額、付款金額、發票金額、上傳狀態。
- 法規未確認前，不得對外承諾完整正式上線。

### 支付

- POS 不保存完整卡號、CVV、磁條資料、PIN block。
- 每筆付款要有 PSP transaction id、授權碼、金額、狀態、對帳欄位。
- 必須處理重複付款、未知狀態、退款、請款失敗、金額不一致。

### 庫存 / BOM

- 庫存用 ledger，不得直接改 balance 當真相。
- 銷售、退貨、報廢、調撥、盤點都必須是 movement。
- BOM 版本要保存，不能讓歷史訂單被新配方改寫。

### AI

- AI 第一版只做摘要、異常提醒、可審核建議。
- 不得自動下單、自動改價、自動發促銷，除非通過 `docs/high-risk-workstreams.md` 的 go/no-go gate。
- AI 建議必須附依據資料、時間窗、信心等級與人工核准。
- 資料不足時必須 abstain，不得硬編理由。

## 6. 任務拆解規則

每張開發票據必須包含：

```text
背景：
目標：
不做範圍：
資料表/API/畫面：
狀態與錯誤：
權限與租戶隔離：
稽核與監控：
測試案例：
手動 QA：
上線/回滾：
待確認事項：
```

不得只寫一句「實作電子發票」、「串 LINE Pay」、「做 AI 備料」。

## 7. 驗收規則

每次交付至少要驗收：

- happy path
- 一個壞輸入
- 一個外部服務失敗
- 一個重試或重送
- 一個權限拒絕
- 一個 tenant 隔離案例
- 一個 audit log 案例
- 一個手動補救案例

文件型交付也要驗收：

- README 是否連到文件
- 文件是否有明確範圍
- 是否列出不做項
- 是否列出 go/no-go gate
- 是否列出待確認人與問題

## 8. 待確認不得偷渡

任何不確定事項都必須用以下格式標示：

```text
待確認：
問題：
確認人：
確認來源：
未確認前禁止：
影響文件/功能：
```

範例：

```text
待確認：電子發票離線補傳期限
問題：斷網交易最晚多久要完成上傳？
確認人：會計師 + 電子發票加值中心
確認來源：財政部規範 / 合約 / 書面回覆
未確認前禁止：對外承諾「完全離線正式開發票」
影響文件/功能：發票模組、銷售頁、POS 離線模式
```

## 9. Definition of Done

工程完成不等於程式寫完。完成必須同時滿足：

- 規格完整
- 程式可執行
- 測試通過
- 手動 QA 通過
- 錯誤與例外可處理
- audit log 可追
- 監控可看
- 回滾可做
- 文件已更新
- 待確認事項沒有被偷渡成既定事實

## 10. 最高否決權

以下任一狀況可以直接擋下開發或上線：

- 用「等等 / 之類」省略高風險流程。
- 沒有 idempotency 就做同步或付款。
- 沒有 tenant scope 就查資料。
- 沒有 audit log 就做發票、付款、退貨、作廢、權限變更。
- 沒有人工補救流程就接外部服務。
- 沒有法規/合作商確認就承諾電子發票或金流正式上線。
- AI 不能說明依據卻輸出營運建議。

這些不是品質建議，是專案紅線。
